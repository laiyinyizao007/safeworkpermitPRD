# ADR-002: 产品范围从单一动火系统升级为完整PTW系统

**日期**：2026-03-09
**状态**：已接受
**决策者**：用户 + Claude Code (Opus 4.6)

---

## 背景与问题

### 初始方案的问题
项目最初定位为"动火作业票电子审批系统"，仅覆盖GB 30871-2022规定的8大特殊作业中的1种（动火作业）。

### 发现的严重缺陷
用户（资深专家）指出这种设计会导致：

1. **法律合规风险**：
   - GB 30871-2022统一规范8大特殊作业，监管部门审计时会核查所有票证
   - 单一系统导致电子票与纸质票混用，管理混乱

2. **交叉作业管理缺失**：
   - 标准明确要求"一票多办"：同一作业涉及多种特殊作业时，应同时办理各自审批手续
   - 单一系统无法处理交叉作业场景（如受限空间内动火）

3. **数据孤岛问题**：
   - 气体分析数据无法在动火、受限空间等作业间共享
   - 人员资质需要重复录入和验证
   - 监护记录无法复用

4. **空间冲突无法规避**：
   - 无法实现基于地理位置的作业冲突检测
   - 例如：动火点下方进行喷漆作业，系统无法识别风险

---

## 决策

**采用"1+8"架构模式，设计完整的PTW（Permit to Work）系统**

### 产品重新定位
- **产品名称**：危险化学品企业特殊作业许可（PTW）管理系统
- **覆盖范围**：GB 30871-2022规定的8大特殊作业
  1. 动火作业
  2. 受限空间作业
  3. 盲板抽堵作业
  4. 高处作业
  5. 吊装作业
  6. 临时用电作业
  7. 动土作业
  8. 断路作业

### "1+8"架构设计

#### "1"：通用底座
- **基础信息管理**：企业组织架构、区域划分、设备台账
- **人员库**：人员信息、资质证书、培训记录、健康档案
- **JSA风险库**：8类作业的标准风险库、企业自定义风险库
- **IoT接口层**：气体检测仪、定位设备、视频监控、环境传感器
- **定位引擎**：地理围栏、空间冲突检测、动态隔离带管理
- **审批引擎**：工作流引擎、权限管理、电子签名、CA认证
- **数据共享层**：气体分析数据、监护记录、安全措施、应急预案

#### "8"：作业票模块
每个模块包含：
- 作业申请表单（符合GB标准）
- 作业分级逻辑（如动火分特级/一级/二级）
- 审批流程配置
- 安全措施清单
- 现场监护要求
- 完工验收标准

---

## 备选方案

### 方案1：保持单一动火系统（已否决）
- **优点**：开发周期短、成本低
- **缺点**：无法满足合规要求、存在管理脱节风险、无法处理交叉作业

### 方案2：先做动火，再逐步扩展（已否决）
- **优点**：MVP快速上线
- **缺点**：架构设计不完整，后期扩展成本高、可能需要推倒重来

### 方案3：完整PTW系统（已采纳）
- **优点**：
  - 100%符合GB 30871-2022合规要求
  - 数据共享，一次录入多处调用
  - 交叉作业冲突规避
  - 空间隔离带管理
  - 统一的监护和验收逻辑
- **缺点**：开发周期较长、初期投入较大
- **应对**：采用分阶段交付策略，MVP先上线核心作业（动火+受限空间），V2.0扩展到8大作业

## 决策理由

### 1. 法律合规的刚性要求
GB 30871-2022是国家强制性标准，监管部门在安全审计或事故调查时，会核查8类作业的审批手续是否齐全。单一系统无法满足合规要求。

### 2. 交叉作业是行业常态
化工现场的"坑"往往出在交叉作业上：
- 受限空间内动火（动火+受限空间）
- 高处动火（动火+高处）
- 罐区吊装（吊装+动火+高处）

系统必须能够：
- 自动识别交叉作业
- 强制关联多张作业票
- 共享气体分析和监护数据
- 基于地理位置检测空间冲突

### 3. 数据共享带来巨大价值
"一次录入、多处调用"：
- **气体分析共享**：受限空间和动火作业都需要分析可燃气体，数据可共享
- **人员资质复用**：同一人员可能涉及多种作业，资质一站式核验
- **监护记录复用**：交叉作业时，监护人的在岗记录可共享

### 4. 政策驱动
应急管理部鼓励企业建设集"重大危险源管理、双重预防机制、特殊作业、人员定位"为一体的智能化管控平台。

## 影响

### 对产品文档的影响
- **PRD**：需要重新编写，覆盖8大作业 + 通用底座
- **Roadmap**：需要调整分阶段策略
- **PROJECTWIKI.md**：需要补充8大作业的业务知识

### 对开发的影响
- **架构设计**：需要设计通用底座和模块化架构
- **开发周期**：MVP阶段聚焦核心作业（动火+受限空间），V2.0扩展到8大作业
- **技术栈**：需要支持模块化、可扩展的架构

### 对用户的影响
- **合规性**：100%符合GB 30871-2022要求
- **易用性**：数据共享，减少重复录入
- **安全性**：交叉作业冲突规避，降低事故风险

## 验证方式

### 合规性验证
- 对照GB 30871-2022标准，能
- 邀请安监专家进行合规性评审

### 业务价值验证
- 试点企业实际使用，验证交叉作业管理效果
- 统计数据共享带来的效率提升

### 技术可行性验证
- 架构设计评审，确保可扩展性
- 性能测试，确保支持8大作业的并发量
