# 06 - 用户工作流

> **本章导读**: 本章详细介绍配置端的完整用户工作流，涵盖设计阶段(Design Phase)、部署阶段(Deployment Phase)和执行阶段(Execution Phase)三个核心阶段。

---

## 6.1 工作流全景图

### 6.1.1 三阶段工作流模型

配置端采用"设计-部署-执行"三阶段工作流模型，实现从表单配置到现场使用的完整闭环。

```mermaid
flowchart LR
    subgraph "Phase 1: Design 设计阶段"
        D1[选择模板] --> D2[配置字段]
        D2 --> D3[编排布局]
        D3 --> D4[设置规则]
        D4 --> D5[预览测试]
    end

    subgraph "Phase 2: Deployment 部署阶段"
        P1[版本管理] --> P2[灰度发布]
        P2 --> P3[权限分配]
        P3 --> P4[上线生效]
    end

    subgraph "Phase 3: Execution 执行阶段"
        E1[扫码拉取] --> E2[动态渲染]
        E2 --> E3[填写提交]
        E3 --> E4[校验拦截]
        E4 --> E5[数据沉淀]
    end

    D5 -->|保存配置| P1
    P4 -->|发布上线| E1

    style D1 fill:#e1f5ff
    style P1 fill:#fff4e6
    style E1 fill:#e8f5e9
```

### 6.1.2 角色与职责

| 角色 | 主要职责 | 使用阶段 | 典型用户 |
|------|---------|---------|---------|
| **配置管理员** | 设计表单、配置规则、发布版本 | Design + Deployment | 安全主管、系统管理员 |
| **审批人员** | 审核配置变更、批准上线 | Deployment | 部门经理、安全总监 |
| **现场操作员** | 填写作业票、执行作业 | Execution | 作业人员、班组长 |
| **监护人员** | 监督作业、确认安全措施 | Execution | 安全员、监护人 |

**规则配置界面设计**:

| 规则类型 | 配置方式 | 用户体验 |
|---------|---------|---------|

| **条件规则** | 可视化条件构建器 | 拖拽字段，选择运算符，输入值 |
| **复杂规则** | 表达式编辑器 | 支持语法高亮、自动补全 |
| **高级规则** | 代码模式(可选) | 面向开发人员的扩展能力 |

## 6.4 Execution Phase（执行阶段）

执行阶段是现场人员使用配置好的表单进行作业票填写和审批的过程。

### 6.4.1 扫码拉取

**表单获取流程**:

```mermaid
sequenceDiagram
    participant User as 现场人员
    participant App as 移动端App
    participant API as 后端API
    participant Cache as Redis缓存
    participant DB as 元数据库

    User->>App: 扫描作业区域二维码
    App->>API: 请求表单配置<br/>(zone_id, permit_type)
    API->>Cache: 查询缓存
    alt 缓存命中
        Cache-->>API: 返回配置
    else 缓存未命中
        API->>DB: 查询元数据
        DB-->>API: 返回配置
        API->>Cache: 写入缓存
    end
    API-->>App: 返回表单配置JSON
    App->>App: 解析并渲染表单
    App-->>User: 显示作业票表单
```

**离线模式支持**:

```mermaid
flowchart TD
    Start[打开App] --> Check{网络状态}

    Check -->|在线| Online[从服务器获取最新配置]
    Check -->|离线| Offline[使用本地缓存配置]

    Online --> Sync[同步到本地缓存]
    Sync --> Render[渲染表单]
    Offline --> Render

    Render --> Fill[填写表单]
    Fill --> Submit{提交}

    Submit -->|在线| DirectSubmit[直接提交到服务器]
    Submit -->|离线| LocalSave[保存到本地队列]
    LocalSave --> WaitNetwork[等待网络恢复]
    WaitNetwork --> AutoSync[自动同步上传]
```

### 6.4.2 动态渲染

**渲染流程**:

```mermaid
flowchart TD
    Config[表单配置JSON] --> Parser[元数据解析器]

    Parser --> Schema[解析Schema层<br/>字段定义]
    Parser --> Layout[解析Layout层<br/>布局配置]
    Parser --> Rules[解析Rules层<br/>业务规则]

    Schema --> Registry[组件注册表]
    Registry --> Components[实例化组件]

    Layout --> Container[布局容器]
    Components --> Container

    Rules --> Expression[表达式引擎]
    Expression --> Bindding[数据绑定]

    Container --> Bindding
    Bindding --> UI[渲染UI界面]
```

**渲染性能优化**:

| 优化策略 | 说明 | 效果 |
|---------|------|------|
| **懒加载** | 非可视区域组件延迟加载 | 首屏渲染时间减少40% |
| **虚拟滚动** | 长列表使用虚拟滚动 | 内存占用减少60% |
| **组件缓存** | 已渲染组件实例缓存 | 切换标签页无需重渲染 |
| **增量更新** | 仅更新变化的组件 | 交互响应时间<100ms |

## 6.6 最佳实践

### 6.6.1 设计阶段最佳实践

1. **从模板开始**: 优先使用标准模板，在此基础上定制
2. **字段命名规范**: 使用有意义的英文key，如`gas_oxygen_level`
3. **分组合理**: 按业务逻辑分组，每组不超过10个字段
4. **规则简化**: 避免过于复杂的条件嵌套，保持可维护性
5. **充分测试**: 在预览模式下测试所有场景

### 6.6.2 部署阶段最佳实践

1. **版本说明清晰**: 每个版本都要有明确的变更说明
2. **灰度发布**: 重大变更必须先灰度，观察无异常再全量
3. **回滚预案**: 发布前准备好回滚方案
4. **权限最小化**: 按需分配权限，避免过度授权

### 6.6.3 执行阶段最佳实践

1. **离线准备**: 进入作业区域前预加载表单配置
2. **及时保存**: 填写过程中定期自动保存草稿
3. **错误处理**: 遇到校验错误时仔细阅读提示信息
4. **数据完整**: 确保所有必填项和签名都已完成
